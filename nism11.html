Team Activity: Debate

Your tutor will split you into 4 teams and then you will debate the following questions:

Team 1: It is our belief that the future of the Internet is based on content centric networking (CCN &/or NDN or COAST).

Group 1: Opening argument 3

The current host-centric Internet architecture was not designed for security, leaving it susceptible to many different types of attacks (My-course.co.uk, 2020).

Content-centric networking (CCN) architectures, such as named data networking (NDN) and content aware searching retrieval and stream (COAST), include security in their design, addressing many of the attack vectors which threaten host-centric networking (Ding, W. et. al., 2016). Security improvements include:

  • CCN largely eliminates the problem of distributed denial of service (DDOS) attacks, which targets hosts rather than data (Bilal, M. et. al., 2020).
  • CCN architectures incorporate security into their design, providing improved confidentiality and integrity controls around data using concepts including secure distribution of protected content (SDPC) (Ding, W. et. al., 2016).
  • Some CCN architectures, including COAST, allow for intelligent overlays which can monitor for suspicious content requests (Ding, W. et. al., 2016).

By solving some of the most vexatious security problems, content-centric networking is the future of the Internet.

Introduction

Loaded With Stuff! (loadedwithstuff.co.uk) is an e-commerce site targeting customers looking for the newest technology from known brands. In an earlier design document, the authors identified the regulatory bodies applicable to the website and reviewed a list of relevant theoretical security vulnerabilities.

In this executive summary, the authors provide an overview of the confirmed theoretical vulnerabilities as well as newly discovered ones applicable to LoadedWithStuff! website, and identifies mitigation recommendations ordered by business priority which address applicable regulations identified in the Design document.

Scanning Methodology

The list of vulnerabilities identified in the design document is not considered all encompassing, thus, to ensure all critical web vulnerabilities have been addressed in this executive report, the OWASP Top 10 (2021) was employed as a framework for performing vulnerability scans. The OWASP recommendations target the most critical exposures and are well documented and used within the industry (Fredj et al, 2021).

By using the OWASP Top 10, a standard security baseline is developed. It ensures the presence of critical security configurations and minimises risk by providing required technical and administrative controls that are implemented as required due diligence, in a proactive approach (Jenkins,2022). The OWASP top 10 provides any business with value through following the processes illustrated in our methodology in Figure 1 (RedScan Teams, 2020).

Limitations of Scanning Activities

The hosting provider for the target website, A2 Web Hosting, has implemented a number of security controls for their clients (A2hosting.com, 2022):

  • Distributed Denial of Service Protection
  • Dual Web Hosting Firewalls
  • Brute Force Defence
  • Virus Scanning
  • Server Hardening
  • Security Monitoring

These controls often interfered with the scanning activities undertaken for this report by generating error messages and CAPTCHA challenges, as shown in Image 1. Access to the site was sometimes blocked too, causing scanning to abort. In order to avoid permanent blocks, tests were carried out using various virtual private network (VPN) services.

We could also not confirm the vulnerabilities introduced by improper log configurations or insecure website design (OWASP TOP 10 2021 categories 4 and 9); this would require additional access to the page’s Admin panel, its code and database. This is out of scope for this executive summary, however, it should be considered when conducting any future assessments as an improvement.

Security Misconfiguration		 3

Scanned with Kali Linux (sqlmap, nikto, skipfish, netcat), Nmap, Nessus

Automated scans identified no significant misconfigurations in the system. While Nmap reported thousands of open ports, Nessus suggested that these are false positives. This may indicate that the server is protected by a Tarpit or similar defence. Tarpit systems slow down and confuse attackers, by responding to requests on any port number (Dalke, 2014).

However, manual testing with Netcat has shown that the server does have some open ports, including File Transfer Protocol (FTP). FTP is considered an insecure protocol as it does not encrypt the login credentials nor the data being transferred (Helpsystems.com, 2020).

Recommendations

Recommendations on this report are based on the security vulnerabilities discussed in Tables 2 and 3. The list has been ordered based on its business priority which correlates to vulnerability DREAD risk rankings.

OWASP category Business Priority Recommendations
A07:2021 High

Introduce password complexity by forcing users to include a mix of numbers, letters and special characters.

Place a limit or increasingly delay failed login attempts. Implement brute force protection and account lockout policy.

Implement Multi Factor authentication (OWASP, 2021).

A05:2021 High Disable the FTP protocol and use SSH File Transfer Protocol (SFTP) instead (Stahnke, 2006).
A06:2021 & A08: 2021 Medium Implement version management, frequently check for patches and upgrade all libraries to the latest version (Zhang et al, 2021).
A07:2021 Medium Update Softaculous to the latest version and implement good patching processes (NVD, 2022).
A08/2021 Low Add a temporary (with short expiry period) unpredictable valid token to any form submission (scanRepeat, 2022).
A02:2021 Low Enable HTTP Strict Transport Security on the server (OWASP, 2012).
A10:2021 Low

Validate all input within the tool to ensure that it meets the business and technical requirements (OWASP, 2022).

Validate incoming IP addresses. Ensure all addresses are either IP V4 or V6.

Ensure all data comes from a valid domain name.

Disallow entry of complete URLs by the user.

A07:2021 Low

Update default username (Bradley, 2017).

Update default password to include lower case letters.

A05:2021 Low

Use of the domain mail ID to send emails.

Train staff to understand the risk of internal phishing, pretexting attacks.

Conclusion

The list of recommendations in Table 3 address a variety of currently relevant web application vulnerabilities; it would counter the most common weaknesses and strengthen the security of the website. However, readers should be aware that no web application can ever be 100% secure (Vigliarolo, 2018), and this list is by no means exhaustive. There may be additional vulnerabilities that could not be detected due to missing accesses to the admin page, code and database or where the detection has been prevented by the A2 Web Hosting security controls.

To further strengthen an e-commerce website, ISO 21001 recommends performing vulnerability scans on a quarterly basis for early threat detection (Stankovic, 2021). The vulnerability scanning framework could also be expanded to look for other common vulnerabilities such as those listed in the National Vulnerability Database (NVD) (Sane, 2020) in addition to the OWASP Top 10. The use of penetration testing should also be considered to detect threats not picked up by the automated scanners.

It is also suggested that anyone handling the website should receive Cyber Security training whilst the organisation should consider implementing a Cyber Security framework to streamline the approach to security threats (Nidecki, 2022).
