Summary Post

In groups and individually, we were asked to perform a preliminary scan of our target websites and review the results from our peers. Among the answers that we needed to find were the nameservers, mail servers, and registration information for the domains.

For the most part, these scans were straight-forward. A typical response was produced by Tolofari (2022) which correctly identified these results for the group 2 target website. The question which unexpectedly ended up being controversial was finding the physical location of the server hosting the site. Every initial post from the other groups which addressed this question placed the server in the USA, often specifically in Ann Arbor, Michigan. This is understandable as most web-based IP address geolocation sites such as geolocation.com and ip2location.com do indeed report the address as belonging to A2 Web Hosting in Ann Arbor. However, our group opted to not take this information as the final word and instead looked closer at our individual traceroutes, reviewing the hops for geographic clues and looking for who appeared to have trans-Atlantic hops and who did not. As a result we finally agreed that the most likely physical location for the server is Amsterdam, Netherlands (Angelides, 2022).

What I found most interesting about this topic was the reaction from others towards our group’s answer to this question. Rather than re-evaluating their own previous conclusions, responders challenged ours. Klacar (2022) argued that IP address geolocation should be considered reliable at least at the country level, and Jurbe (2022) suggested that because the autonomous system number (ASN) associated with the target network block is assigned to A2 Web Hosting in the USA, the site is most likely in the USA.

While this debate is something of a teapot-based tempest, I believe that it has some important lessons. Cybercriminals routinely use stealth technology such as Tor, the dark web, and overlay networks to evade detection and attribution (Campbell, 2016). To succeed against these deceptive threats, Sanders (2016) argues that curiosity, and the desire to uncover the truth beyond initial appearances, is a critical factor for success in cybersecurity.

Introduction

Loaded With Stuff! (loadedwithstuff.co.uk) is an e-commerce site offering two payment options: outright purchases and ‘Easy Financing’ program. It targets customers looking for the newest technology from known brands. In this report, the authors will describe the system, identify the security and privacy compliance requirements which apply to it, and provide a prioritised list of potential vulnerabilities and recommended mitigations.

System Description

Loaded With Stuff! was built with the Softaculous automated deployment system and is based on Loaded Commerce Community Edition v6.6. It is located at the A2 Web Hosting data centre in Amsterdam on a shared hosting server with IP address 68.66.247.187. The loadedwithstuff.co.uk domain name is registered through eNom LLC.

The website collects and stores data, some of which is considered personal, to be able to process the orders, such as: email address, delivery and billing addresses, phone numbers.

The website does not handle or store any cardholder data (CHD) and the payment page has instead been outsourced to a third party.

The ‘Easy financing’ finance program appears to be outsourced to an external provider and, thus, it is out of scope of this design document.

Security Vulnerabilities & Mitigations

The theoretical security vulnerabilities relevant to the above list of identified regulations have been prepared using STRIDE methodology and can be viewed below, where each vulnerability has been assigned to a STRIDE (Microsoft, 2022) category (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege) and ordered by business priority. The recommendations have also been prepared for each identified security weakness.

The hosting provider for the site, A2, provides a number of common security mitigations to all customers, including DDoS protection, firewalls, brute force defense, and virus scanning (A2 Web Hosting, 2022). These mitigations are not included in the table below as they are assumed to already be implemented. Business priority has been determined based on each vulnerability’s DREAD risk ranking (University of Essex Online, 2022):

  • DREAD 0 (no risk), 1 (low risk) - low business priority.
  • DREAD 2 (medium risk) - medium business priority.
  • DREAD 3 (high risk) - high business priority.

STRIDE category Business priority Vulnerability description Recommendation
Elevation of Privilege High GDPR, DPA. Weak default Softaculous admin password and username. When installing a new site, Softacolous sets default username to ‘Admin’ and the password is generated with capital letters, numbers and special characters (AuthorityRush, 2018). Update default username. Update default password to include lower case letters.
Tampering, Denial of Service High PCI. Several open ports exposed on server (Lee, 2022). Close all not needed ports (Gelnaw, 2019). Switch to secure protocol formats (example, FTP to be switched to SFTP). Deploy the site behind a web application firewall (WAF).
Tampering High GDPR, PCI. A06:2021-Vulnerable and Outdated Components Site uses Loaded Commerce Community Edition 6.6, which does not appear to have been updated since 2018. Software is no longer being maintained, and Google (2022) considers the Loaded Commerce site to be “dangerous and deceptive”. Use an e-Commerce software product that is up to date and maintained by its vendor. Implement good patching processes (OWASP, 2022a).
Tampering, Elevation of Privilege Medium PCI. Potential web server operating system exploits (Jackson, 2009). Follow a patch management process. Utilise a patch monitoring agent (manageEngine, 2022).
Elevation of Privilege Medium GDPR, PCI. CVE-2020-26886: Local Privilege Escalation using softaculous/bin/soft. Site was deployed using Softaculous script and may be vulnerable to CVE-2020-26886. Update Softaculous to the latest version and implement good patching processes (NVD, 2022).
Tampering, Spoofing, Repudiation Medium GDPR, DPA. Cross-Site Request Forgery (CSFR). The features allowing users to update their details and reset passwords may be vulnerable to this type of attack (OWASP, 2022b). Ensure the current website framework uses the latest Softaculous version which includes CSRF protection. Ensure the CRSF check is enabled by the ISP manager for the website (Mittal, 2019).
Information Disclosure Medium GDPR, PCI. CVE-2014-5140. Site may be vulnerable to SQL injection (Mitre, 2014). Web application firewall is already deployed as part of A2 Web Hosting’s security package. Implement penetration testing to determine if the site is vulnerable (Breaking Technology, 2014).
Spoofing Medium GDPR,PCI. Password policy is too weak. It allows sequential, repeated passwords and dictionary words. User accounts are not locked out after more than 10 failed login attempts. No multi factor authentication is enforced. Site will be vulnerable to a Brute force attack. Enforce strong password policy (Davin, 2020).
Tampering Medium GDPR, PCI. A06-2021- Vulnerable and outdated components: Site uses PHP 7.3.33 which is end of life and does not support new security fixes. Upgrade the software (php.net/eol.php, 2022).
Repudiation Low The correct logs must be set up for the website in the Softaculous Admin Panel (Musukama, 2022).
Spoofing / Information Disclosure Low GDPR. Email Phishing. The site does not use the domain to generate emails: the email ID is from a different domain. Use of the domain mail ID (example Support@loadedwithstuff.co.uk) instead of the current account that sends the registration information. Train staff to understand the risk of internal phishing, pretexting attacks. Implement spoofing protection mechanisms such as SPF or DMARC (Dmarcly, 2022).

Conclusion

E-commerce sites remain one of the most highly prized targets for cyber threats, with security incidents growing by 20% in 2021 (Webscale, 2021). In this paper, the authors have listed a number of potential security vulnerabilities faced by the Loaded With Stuff! Website. The paper further provides a prioritized list of mitigations which the operators of the site should consider in order to secure their site against cyber threats while maintaining compliance with their legal and regulatory obligations.
